# Toolchain: Uca Orchestration & Build Systems

The Uca toolchain is a deterministic, state-aware orchestrator engineered to maintain strict parity between the declarative `uca.yaml` manifest and the application's physical source artifacts.

---

## 1. CLI Foundation: Lifecycle Orchestration

The `uca` CLI serves as the authoritative interface for environment provisioning, iterative development, and production-grade artifact exportation. It enforces mandatory static analysis and type validation at every phase of the application lifecycle.

### Primary Lifecycle Stages
- **`init <app-name>`**: Provisions the hermetic project structure and initializes the root level `uca.yaml` manifest.
- **`dev`**: Spawns the concurrent development runtime, integrating multi-layer Hot Module Reloading (HMR) for `pages/`, `services/`, and `agents/`.
- **`tidy`**: The core reconciliation engine. It parses `uca.yaml` for structural changes, executes database migrations, resolves dependencies, and rebuilds cross-runtime type definitions.
- **`test`**: Orchestrates a unified testing pipeline, executing unit and integration suites across the Go, TypeScript, and Python layers.
- **`export <binary>`**: Compiles the polyglot stack into a highly optimized, immutable `.tar.zst` OCI-compliant appliance.

### Hierarchical Administrative Interface
The CLI utilizes a granular command hierarchy for precise component management:

| Functional Domain | Management Commands |
| :--- | :--- |
| **System Config** | `icon <path>`, `port <name> <val>`, `version` |
| **View Layer** | `add`, `remove`, `deps`, `test`, `list`, `info` |
| **Logic Services** | `add <name> <methods>`, `remove`, `deps`, `test`, `list`, `info` |
| **Agent Runtimes** | `add <name> <model>`, `remove`, `deps`, `test`, `list`, `info` |
| **Persistence** | `list`, `schema`, `seed <file>`, `reset <table>` |
| **Authorization** | `add`, `remove`, `list` |
| **Environment** | `add <name> <value>`, `remove`, `info` |
| **Observability** | `logs {list\|info\|clear}`, `tests {add\|remove\|list}` |

## 2. Dependency Architecture: Hermetic Runtime Provisioning

Uca eliminates "standard-environment" variance through a strict **Downloader Design Pattern**, ensuring that every developer and production instance utilizes bit-identical toolchain versions.

### Environment Isolation
The CLI autonomously fetches and encapsulates specialized versions of the Go compiler, Node.js runtime, and Python interpreter within the local `.uca/` directory.

### Deterministic Dependency Resolution
- **`deps.lock`**: An immutable resolution manifest updated exclusively via the `tidy` process.
- **System Constraints**: `uca deps add <pkg>` persists requirements to `sysdeps.txt` for consistent environment setup.
- **Bulk Ingestion**: Supports manifest-driven updates via `uca deps add -f <path-to-deps.lock>`.
- **Native Reconciliation**: Discovered dependencies are automatically synchronized into language-native formats (`go.mod`, `package.json`, `requirements.txt`) based on the target runtime.

## 3. The Interop Bus: Cross-Runtime Static Type Reconciliation

The toolchain bridges the Logic, View, and Agent execution layers through automated code generation and static analysis.

### End-to-End Type Integrity
During the `dev` or `tidy` cycles, the CLI performs introspection of Golang structures within `services/` to synthesize corresponding **TypeScript interfaces**. This ensures compile-time type safety from the persistence layer to the client-side presentation.

### The Uca Registry Map
The CLI maintains an authoritative **Registry Matrix** to validate cross-layer interoperability:
- **Loaders**: Ensures `pages: loader:` definitions map to active, verified Go service implementations.
- **Tool-Calling**: Automatically validates that `agents: tools:` references conform to the required service method signatures.

### Environment Abstraction
- **Key Definitions**: `uca.yaml` maintains a manifest of required environment keys.
- **Secure Persistence**: Secret values reside in a VCS-ignored `.env` file within `.uca/`.
- **Runtime Injection**: Secrets are injected at the project root level, ensuring secure, uniform access across all polyglot runtimes.

---

## 4. Observability & Validation

- **Unified Testing Interface**: `uca tests` provides a standardized entry point for managing heterogeneous test cases.
- **Agent Schema Validation**: The toolchain enforces that any service function exposed as an AI tool includes a mandatory `desc:` (description) metadata field, fulfilling the strict schema requirements of `smolagents`.
- **Telemetry Management**: Aggregated diagnostic logs are unified into a single stream, manageable via `uca logs` utility commands.
